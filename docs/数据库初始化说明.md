# 数据库初始化说明

## 1. 当前状态

当前代码已经切换到新的 MySQL 表结构，成员资料会先从 `config/members.json` 同步到 `members` 表，然后消息再写入 `messages` 表。

说明：配置文件里仍然保留外部数据源的 `rank` 字段名，但写入数据库时会落到 `members.election_rank`，避免使用 SQL 保留字。

因为 `messages.server_id` 依赖 `members.server_id` 外键，所以 `config/members.json` 必须先提供完整成员资料，至少要包含以下字段：

```json
{
  "id": 417331,
  "ownerName": "杨媛媛",
  "serverId": 951577,
  "channelId": 1312655
}
```

## 2. 当前 config/members.json 要求

当前 `config/members.json` 已经升级到新版完整结构。

新结构下至少需要：

- `id`
- `ownerName`
- `serverId`
- `channelId`

如果缺少这些关键字段：

- `sync_members()` 会跳过这些不完整成员
- `members` 表不会写入这些成员
- 后续 `messages` 写入会因为 `server_id` 外键失败

## 4. 初始化步骤

### 第一步：备份旧数据库

如果你旧库里还有数据，先自己备份。

### 第二步：重建数据库表

执行：

- `docs/数据库建表语句.sql`

建议先清空旧表，再执行新的建表语句。

### 第三步：补全 config/members.json

必须至少给每个成员补上：

- `id`
- `ownerName`
- `serverId`
- `channelId`

### 第四步：运行一次单次抓取

例如：

```bash
python src/pocket48_scraper.py -c config/config.json --once
```

程序启动时会自动先同步成员资料到 `members` 表。

### 第五步：检查数据库

重点检查：

- `members` 是否已有成员资料
- `messages` 是否正常写入
- `crawl_tasks` / `crawl_checkpoints` 是否正常生成

## 5. 建议检查 SQL

### 检查 members

```sql
SELECT id, owner_name, server_id, channel_id
FROM members
ORDER BY id DESC;
```

### 检查 messages

```sql
SELECT message_id, room_id, server_id, channel_id, sender_user_id, sender_name, message_type, message_time
FROM messages
ORDER BY message_time DESC
LIMIT 20;
```

### 检查抓取断点

```sql
SELECT server_id, channel_id, last_message_id, last_message_time_ms, last_success_at
FROM crawl_checkpoints;
```

## 6. 当前 members.json 是否可直接使用

可以直接使用。

原因：

- 当前 `config/members.json` 已补全 `id`
- 当前 `config/members.json` 已补全 `ownerName`
- 现有成员已经满足新结构最小入库要求

## 7. 推荐后续做法

最稳的方案是把 `config/members.json` 直接升级成接近 `yaya_msg` 的完整格式，后续新增成员就只需要复制一段成员资料进去。
